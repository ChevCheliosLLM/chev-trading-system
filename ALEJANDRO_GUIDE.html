02 — The Modules

What each file actually does

Think of each file as a department in a trading firm. Each has one job. Each has rules about who can touch it.

⚙️
engines.py
The Calculator — ~3,000 lines

This is the pure mathematics of the system. It does not make decisions. It only measures. It receives raw candle data and outputs structured numbers that Dexter uses to build his analysis. If you break engines.py, Dexter stops working entirely.

Input
  • OHLCV candles (price data)
  • Timeframe (1H, 4H, etc.)
Output
  • Swing highs/lows
  • Leg strength scores
  • SR zone coordinates
  • Fibonacci anchors
  • Volume Profile bins
  • Pattern signals
⚠️ Alejandro: coordinate with Kev before editing
🔬
dexter.py
The Research Analyst — ~3,000 lines

Dexter runs on a 5-minute loop. He calls engines.py to get the maths, then scores each setup, counts confluences, and decides if Chev should hear about it. He writes everything to Google Sheets and builds the report that goes to Chev. He has no opinion — only measurements.

Input
  • engines.py scores
  • Watchlist of pairs
  • Timeframe configs
Output
  • Confluence count + list
  • Structured report for Chev
  • Google Sheets rows
  • Trade alerts via Telegram
⚠️ Alejandro: coordinate with Kev before editing
🧠
Chev (LLM in Open WebUI)
The Senior Trader — AI decision maker

Chev is a large language model (Claude) running inside Open WebUI with a detailed system prompt. He reads Dexter's structured report and decides: trade or skip? He has personality, strategy, and risk awareness baked into his system prompt. He posts trade ideas to Telegram and the webapp.

Input
  • Dexter's analysis report
  • Jane's signals (optional)
  • His own system prompt
Output
  • Trade idea (pair, direction, SL, TP)
  • Reasoning in plain English
  • Telegram message
⚠️ Both: agree on system prompt changes before making them
🔮
Jane (telegram_listener.py)

Jane is Alejandro's custom analyst. She has her own market view and communicates through Telegram. Her signals feed into the same Telegram channel where Chev operates, giving a second independent perspective before a trade is posted. This is Alejandro's zone — he owns this code.

A single-page web app that shows everything visually: live charts with indicators, open trade ideas, Dexter's confluence results, and Chev's reasoning. It only reads data — it never changes Dexter, engines, or Chev. Kev owns this entirely.

Input

Google Sheets
The Trade Journal — permanent record

Every trade idea, every result, every confluence score gets written here by Dexter. The columns are fixed by index number — column 0, column 1, column 17. If you add or move a column without telling Kev, the entire system reads the wrong data. This is the most fragile part of the whole setup.

Writes
  • Pair, direction, score
  • SL / TP levels
  • Confluences hit
  • Entry / exit times
Reads
  • Performance stats
  • Win rate / drawdown
  • Historical trade log
🚫 Never add/move columns without telling Kev first
03 — The Scoring System

How Dexter scores a setup

This is the heart of the system. Dexter doesn't guess — he counts signals. The more signals that agree, the higher the score. When the score passes a threshold, Chev gets told about it. This is what you need to understand deeply before touching engines.py or dexter.py.

1–2
Confluences
Skip — too weak
2
Confluences
Watch list only
3+
Confluences
Escalate to Chev
Support / Resistance
SR
Price previously bounced from this level. engines.py measures how many times it was tested and how strongly it held. A level tested 8 times is much stronger than one tested twice. This is the most common confluence in the system.
Weight
High
Fibonacci Level
FIB
Based on the first significant move (impulse leg) of the current trend. engines.py identifies where that move started and ended, then projects the standard ratios: 0.382, 0.5, 0.618, 0.786. Price respecting a Fib level is a confirmation of structure.
Weight
Med-High
Volume Profile
VP
A histogram of where the most trading volume happened over the current move. The highest volume bar is the Point of Control (POC) — the price level the market spent the most time at. When price returns to the POC, it often reacts strongly.
Weight
Medium
RSI Divergence
RSI_DIV
Price makes a new high, but RSI makes a lower high (or vice versa for lows). This "divergence" means the move is losing momentum — the engine draws two trendlines to confirm it. When RSI diverges at a key structure level, it becomes a powerful signal.
Weight
Medium
Pattern Recognition
PATTERN
engines.py and patterns.py scan for structured chart formations: compression zones (tight price range before breakout), bull/bear flags (pullback inside a trend), triangles, and channels. Each detected pattern contributes to the score.
Weight
Medium
Swing / Leg Structure
SWING
engines.py breaks price movement into "legs" — each discrete impulse or corrective move. It measures how strong each leg was (range × volume). When price is at the start of a fresh leg at a structure level, the swing context adds confirmation to the setup.
Weight
Supporting
04 — A Real Example

What Dexter's report looks like

This is the kind of structured data Dexter sends to Chev. Every field has a specific source. Understanding this stops you from accidentally breaking a field by changing the function that generates it.

Sample Dexter Analysis Report — BTC/USDT 1H
pair: "BTCUSDT" | tf: "1H"
→
The pair and timeframe being analysed. Dexter loops through the watchlist and runs this for each one.
confluences: 4 | score: 0.78
→
This is the trigger. 4 signals agree on this level. Score of 0.78 out of 1.0. Because confluences ≥ 3, Dexter escalates this to Chev right now.
confluence_list: ["SR", "FIB_0.618", "VP_POC", "RSI_DIV_T1"]
→
Exactly which signals fired. SR level + Fibonacci 61.8% + Volume Profile point of control + RSI divergence trendline 1. The webapp uses this to draw each one on the chart.
direction: "LONG" | price: 67,450
→
Dexter detected structure pointing up and price is sitting at a support confluence. Direction comes from the swing leg analysis in engines.py.
sl: 66,800 | tp1: 68,200 | tp2: 69,100
→
Stop Loss is placed below the structure zone using ATR (Average True Range — a measure of how volatile price is). TP levels are the next SR zones above entry.
leg_strength: 0.82 | auction_zone: [67200, 67650]
→
The strength of the most recent impulse leg (0.82 = strong move, high volume, wide range). Auction zone is where the market balanced before the move — a high-value area according to the VP engine.
05 — What You Can and Can't Touch

Alejandro's rules of engagement

This isn't about trust — it's about not accidentally breaking a live trading system. These rules exist because everything is connected. A change in engines.py can silently break Dexter's confluence count without throwing an error.

File / Area What You Can Do
telegram_listener.py
Your file		 Free Build, break, improve — this is yours entirely. Jane lives here.
Backtesting scripts
New files		 Free Create new .py files for backtesting. Never modify existing files to add testing hooks — write standalone scripts that import what they need.
engines.py
Shared — coordinate		 Coordinate Tell Kev before you edit. Never rename or remove a function — dexter.py calls them by name. Add new functions rather than replacing old ones when improving logic.
dexter.py
Shared — coordinate		 Coordinate Tell Kev first. Only edit when Kev confirms Dexter is not running a live session. Test on your local copy before pushing.
Google Sheets columns
Never without asking		 Never Columns are read by index number throughout the code. Adding a column between existing ones will silently corrupt all historical data reads. Always ask first.
webapp/index.html
Kev's file		 Kev only Leave it alone unless Kev specifically pulls you in. It's a complex single-file app — small changes have cascading effects.
google_credentials.json
Never commit		 Never This file must never appear in a git commit. It's in .gitignore but double-check before every push. It gives full Google Sheets access.
06 — Alejandro's Task List

What to build, in order

These are Alejandro's tasks ranked by priority. Do them in order — the later ones depend on understanding you'll build from the earlier ones.

# Task Why It Matters
1 Set up local environment
Clone repo → pip install → get google_credentials.json from Kev → run python dexter.py		 You can't test anything if you can't run it. Get the system running on your machine first.
2 Backtesting framework
Run the full engine stack against 12 months of historical candle data. Would the strategy have made money?		 Right now we're flying blind. Without historical validation, all improvements are guesses.
3 Improve confluence scoring
An SR level tested 10 times is not the same as one tested 2 times. Make the weights reflect this.		 The biggest signal quality improvement available without changing the strategy architecture.
4 Multi-timeframe gate
Only escalate to Chev when 1H and 4H agree on direction. Dexter currently operates per-timeframe independently.		 Removes a large number of false setups. The best trades always have both timeframes aligned.
5 Fix Fib anchor in engines.py
_impulse_anchors() should return the FIRST leg of the move, not the most recent swing. Look at _find_last_impulse().		 Broken Fib levels mean the confluence is triggering at the wrong price. Both Kev and you are affected by this bug.
6 RSI Divergence → Google Sheets col 17
Make sure RSI_DIV_T1 and RSI_DIV_T2 are always written when RSI divergence is a confluence. Kev needs these to draw trendlines on the chart.		 The webapp cannot draw RSI divergence trendlines until these values reliably appear in the sheet.
7 Stop Loss audit
Look at the last 20 losses. Was SL too tight (ATR multiplier too small), or were the trades genuinely wrong setups?		 One of the fastest ways to improve win rate without changing the entry criteria at all.
8 Jane performance tracking
Jane is live but her win rate isn't tracked alongside Chev's. Add Jane's calls to the performance log.		 We can't compare Chev vs Jane or see if the disagreement flag would be useful without the data.
