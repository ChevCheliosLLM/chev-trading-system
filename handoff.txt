## HOW DEXTER WORKS
================================================================================

Every 2 minutes Dexter scans all pairs across timeframes:
- Crypto: 5m/15m/30m/1h/4h (full Tier A/B SR model, Binance free)
- Forex/Stocks: 15m/30m/1h/4h (Tier B only — protects Twelve Data API budget)

When 3+ confluences align on a pair/TF, Dexter calls Chev via Open WebUI API:
  POST http://localhost:3000/api/chat/completions
  model: "chev-chelios"
  message: [full market brief + candle data + SR levels + Chev's playbook + journal]

Chev replies in strict format:
  POST: <15-word max summary>
  TRADE: direction=long, entry=X, sl=Y, tp=Z, risk_pct=N, leverage=N,
         position_size_usd=N, tags=sr_4h,fib_1h,rsi, trade_type=day,
         sr_level=X, fib_high=X, fib_low=X, vp_start=YYYY-MM-DD,
         rsi_div_t1=YYYY-MM-DD, rsi_div_t2=YYYY-MM-DD
  REASONING: <full analysis, no word limit>

  OR: SKIP: <one sentence why>

Dexter parses the TRADE: line, validates confluence score, checks SL direction,
logs to Google Sheets, and monitors until TP/SL/expiry.

--- DEXTER STARTUP HANG ---

BUG: Dexter froze at "Dexter is watching the markets..." on startup.
  Root cause: connect_to_sheet() called synchronously before Flask starts. Google Sheets
  API was hanging indefinitely (no timeout in gspread). Website unreachable until Sheets
  connected or failed.
  Fix: Moved Flask thread start BEFORE the Google Sheets connection block. Sheets now
  connects with a 3-attempt retry loop (5s sleep between attempts). If all 3 fail,
  Dexter logs a WARNING and continues without Sheets — trade logging disabled but
  website fully accessible.

WHAT DIDN'T WORK — client.session.timeout:
  Tried adding `client.session.timeout = 20` inside connect_to_sheet() to enforce a
  20-second timeout on the gspread HTTP session. This caused:
    AttributeError: 'Client' object has no attribute 'session'
  The gspread version in this project does NOT have a .session attribute on the Client
  object. NEVER add that line. Remove it if you ever see it there.
  Solution was the retry loop + Flask-first approach, not a timeout.

--- api_pending CRASH ---

BUG: "Dexter unreachable" on the website after Dexter started without Sheets.
  Root cause: /api/pending called worksheet.get_all_values() without checking if
  worksheet was None (happens when Sheets connection fails). Caused an AttributeError
  that crashed the Flask request.
  Fix: Added guard at top of api_pending:
    if worksheet is None: return jsonify([])
  Applied same guard to any other endpoint that uses worksheet directly.

--- FIREBASE SSE RACE CONDITION (most complex bug of the session) ---

BUG: RSI div trendlines not appearing when clicking a TF in the RSI÷ dropdown.
  Also: dropdown disappeared after clicking a TF.

  ROOT CAUSE (critical — understand this before touching RSI div code):
  Every call to saveDrawings() does a Firebase PUT. This triggers the SSE stream
  (_subscribeDrawings listener). The SSE listener replaced drawings[] with whatever
  Firebase returned — a version filtered to exclude _rsi_div. So:

    Click TF → _clearRsiDivOverlay() → saveDrawings() → Firebase PUT → SSE fires →
    drawings[] replaced (no _rsi_div) → _drawRsiDivVisual() runs → pushes _rsi_div
    lines to drawings[] → ANOTHER SSE fires (or first one arrives late) → drawings[]
    replaced again → _rsi_div lines wiped → chart redraws with no lines.

  The window between "lines pushed" and "SSE overwrites them" was so short the lines
  never appeared visually.

FIXES APPLIED (all in index.html):

  Fix 1 — Remove saveDrawings() from _clearRsiDivOverlay():
    The old code called saveDrawings() at the end of _clearRsiDivOverlay(). Since
    _rsi_div drawings are never persisted anyway, there's nothing new to save.
    Calling saveDrawings() here was the trigger that fired the SSE race.
    RULE: _clearRsiDivOverlay() must NEVER call saveDrawings().

  Fix 2 — Remove saveDrawings() from _drawRsiDivVisual():
    Old version called saveDrawings() after pushing _rsi_div lines. Same problem.
    RULE: _drawRsiDivVisual() must NEVER call saveDrawings().

  Fix 3 — SSE listener preserves live _rsi_div drawings on Firebase update:
    In _subscribeDrawings, when an SSE message arrives, the listener now does:
      const live_rsi = drawings.filter(d => d._rsi_div);
      drawings = Array.isArray(msg.data) ? msg.data : [];
      if (live_rsi.length) drawings.push(...live_rsi);
    This means even if an SSE fires while RSI div lines are on screen, they survive.

--- DRAWING TOOLS INVISIBLE WHEN RSI PANEL OPEN ---

BUG: Drawing tools (trendlines, hlines, etc.) appeared invisible after placing them
  while the RSI panel was open. Drawings saved fine but weren't visible until refresh.
  Root cause: Two LightweightCharts instances (main chart + RSI chart) both attach
  document-level mousemove handlers. RSI chart's handlers stole the mousemove events,
  preventing chartWrap.mousemove → markDirty() from firing after a drawing was committed.
  Without markDirty(), the canvas never repainted with the new drawing.
  Fix: Added explicit markDirty() call to every drawing commit path in the
  chartWrap click handler (hline, channel, triangle, and the generic trendline/fib
  commit path). Now repaint happens regardless of whether mousemove fires.
