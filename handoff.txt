## HOW DEXTER WORKS
================================================================================

Every 2 minutes Dexter scans all pairs across timeframes:
- Crypto: 5m/15m/30m/1h/4h (full Tier A/B SR model, Binance free)
- Forex/Stocks: 15m/30m/1h/4h (Tier B only — protects Twelve Data API budget)

When 3+ confluences align on a pair/TF, Dexter calls Chev via Open WebUI API:
  POST http://localhost:3000/api/chat/completions
  model: "chev-chelios"
  message: [full market brief + candle data + SR levels + Chev's playbook + journal]

Chev replies in strict format:
  POST: <15-word max summary>
  TRADE: direction=long, entry=X, sl=Y, tp=Z, risk_pct=N, leverage=N,
         position_size_usd=N, tags=sr_4h,fib_1h,rsi, trade_type=day,
         sr_level=X, fib_high=X, fib_low=X, vp_start=YYYY-MM-DD,
         rsi_div_t1=YYYY-MM-DD, rsi_div_t2=YYYY-MM-DD
  REASONING: <full analysis, no word limit>
     Must include: 4H STRUCTURE / INVALIDATION / CONFIRMATION blocks

  OR: SKIP: <one sentence why — must state what was missing>

Dexter parses the TRADE: line, validates confluence score, checks SL direction,
logs to Google Sheets, and monitors until TP/SL/expiry.

Gate order (2026-07-04 — a whole pre-Chev stage was added; see CONFLUENCE SCORING
SYSTEM → TRADEABILITY PRE-GATE for full detail):
  0. TRADEABILITY PRE-GATE (BEFORE Chev is asked): trade threshold, struct pre-gate,
     distance-from-level, directional-conflict — all shadow-logged, all skip Chev
     entirely if failed
  1. Format check (POST:/SKIP: present) — retry once if missing
  2. Post-Chev hard score gate (Dexter's own score vs _active_thresholds(), independent
     of anything Chev tagged — rejects if Chev somehow posts on a setup that shouldn't
     have reached him)
  3. STRUCT_REJECT gate (sr_score ≥ 2 OR vp_score ≥ 2 required) — legacy, redundant with
     the pre-gate's struct check on purpose, both measured
  4. risk_gauntlet validation (ATR SL sanity, R:R ≥ 2.0, position size cap)
  5. Google Sheets log + monitoring loop

## CONFLUENCE SCORING SYSTEM (recalibrated 2026-07-03, thresholds reworked 2026-07-04)
================================================================================

NORMAL minimum score to open a trade: crypto=10, forex=8, stock=8
EXPLORATION minimum (currently ACTIVE — see EXPLORATION MODE section): crypto=7, forex=6, stock=6

Both sets of numbers live as constants near the top of dexter.py
(CONFLUENCE_THRESHOLD_CRYPTO/FOREX/STOCK and EXPLORATION_THRESHOLD_CRYPTO/FOREX/STOCK).
EVERY gate that checks "is this score/distance good enough" MUST read them through the
single helper _active_thresholds(asset_type) — never the raw constants directly — so
exploration mode can never be half-applied across gates. Three call sites currently use
it: the TRADEABILITY PRE-GATE, the post-Chev hard score gate, and the "Score X vs
threshold Y" line shown to Chev in ask_chev_to_judge. _setup_grade() deliberately does
NOT use the helper — grading always uses the FULL normal threshold regardless of
exploration mode, because the grade (and its suggested risk %) is the real risk control
during the exploration phase. See EXPLORATION MODE section for why this matters.

Score = raw_score × TF_multiplier. TF multipliers:
  4H × 1.2   |   1H × 1.0   |   30M × 0.9   |   15M × 0.8

Raw scores by signal type:
  SR (multi-touch, ≥3x):   3pt  — exposed as sr_score
  SR (single/double, 1-2x): 2pt
    NOTE (2026-07-04): if BOTH a resistance and support zone are within the 1.5% window,
    only the NEARER one scores — price sandwiched mid-range no longer collects points
    for both sides (that was chop, not confluence).
  Golden Pocket (GP):       3pt
  Volume Profile (VP POC/VAH/VAL): 3pt — exposed as vp_score
    NOTE (2026-07-04): VP now respects the underlying auction anchor's own validation
    stamps (from _validate_anchor). If the anchor is no longer "active" (price broke back
    through it), that timeframe's VP is skipped entirely. If active but NOT "confirmed"
    (no follow-through yet), the VP still scores but at HALF points, and the reason string
    gets an "unconfirmed-anchor," note — e.g. "VP POC 4h (0.12% away ≤0.40% prox,
    unconfirmed-anchor, 1.5pt)".
  RSI divergence confirmed: 3pt (4pt if both regular + hidden)
    NOTE (2026-07-04): only the two HIGHEST-scoring divergences that share the SAME bias
    (bull or bear) can both score — a Regular Bullish and a Hidden Bearish no longer both
    add points to one setup (that was contradiction counted as confluence; caught live on
    USD/CAD 15m). The dominant-bias divergence always scores; a second one only scores if
    its bias matches.
  EMA (55/21/13 support/resist): 2pt / 1pt / 0.5pt respectively; total EMA capped 3pt
  Bollinger Band (BURST/near/mid): 2pt / 0.5pt / 1pt
    SQUEEZE — NO LONGER SCORED (2026-07-04): demoted to 0pt context, same treatment as
    RSI 50-cross. Every playbook and Chev's prompt said "do NOT enter during a squeeze"
    while the score argued FOR entry — contradiction resolved by making it context-only.
    Reason string still appears (`BB squeeze (width X%, context, 0pt)`) so Chev/Examiner
    still see it, it just can't push a setup over threshold.
  Derivatives — funding + OI trend (crypto only, via derivs.py): 0–4.5pt combined
    "Funding extreme" (contrarian — crowded side is squeeze fuel): 2pt notable / 3pt extreme
    "OI divergence" (price up + OI down = weak rally; price down + OI down = short
      covering): 1.5pt.  "OI confirmation" (price and OI agree — real participation): 1pt.
    See DERIVATIVES LAYER section below.
  Chart pattern (high/mid conf): 2pt / 1pt
  Sweep (liquidity sweep):  3pt
  Fib retracement level:    1.5pt
  RSI OB/OS:                0.5pt
  RSI 50-cross:             context only (0pt — shown in brief, not scored)
  [WATCH — forming div]:    context only (0pt — never scored, shown as amber watch)

TRADEABILITY PRE-GATE (added 2026-07-04, runs BEFORE Chev is ever asked):
  Fires inside scan_pair_tf's caller, in this order, each shadow-logged to the Examiner
  under a distinct NOT_ESCALATED reason so skip_reason_audit() can measure each gate:
    1. Score < _active_thresholds() trade threshold → "below_trade_threshold"
    2. sr_score < 2 AND vp_score < 2 (no structural anchor) → "struct_pregate"

       (a VP anchor IS its own location — SR/Fib distance is irrelevant to it)
       → "too_far_from_level"
    4. Conflicting directional signals: _directional_split(reasons) splits bull vs bear
       points; if the minor side ≥ 2.0 pts AND the dominant side < 2.0× the minor side →
       "conflicting_signals" (contradictory setup, not scored as confluence)

  of 2026-07-04 (commented out in place, not deleted) — this is a paper account, so thin
  weekend books / sweep risk don't cost anything real yet. A REMINDER comment sits right

  real funds are wired in and weekend slippage/sweeps actually matter. This is
  independent of EXPLORATION_MODE now — previously it was gated to `not EXPLORATION_MODE`,

This pre-gate is separate from, and runs in addition to, the OLD structural gate
(STRUCT_REJECT, described below) which still fires after Chev replies via risk_gauntlet's
sibling checks — the two are redundant on purpose right now (measuring both).

Structural prerequisite gate (STRUCT_REJECT, legacy, still present post-Chev):
  sr_score ≥ 2 OR vp_score ≥ 2 must be true before any trade passes.
  Purpose: kills mid-range EMA/fib-only setups with no structural anchor.
  If gate fails: reason_code = "STRUCT_REJECT", NOT_ESCALATED hook fires.

MTF confidence tax (counter-trend trades only):

## EXPLORATION MODE — CURRENTLY ACTIVE (added 2026-07-04)
================================================================================

WHY THIS EXISTS: after the 2026-07-04 scoring recalibration, Dexter's normal threshold
was so tight that Chev received hundreds of escalations over ~10 hours and posted ZERO
trades. With no trades, the learning loops (playbooks, the Examiner's POST-vs-SKIP
comparison) have no data. This is a PAPER account, so a deliberate, time-boxed, fully
reversible exploration phase was built: lower the entry bar, keep every safety/hygiene
gate at full strength, let small graded trades flow, and let the data reveal where the
real threshold belongs.

THE SWITCH — everything hangs off ONE boolean, near the top of dexter.py:
  EXPLORATION_MODE = True     ← flip to False to restore normal thresholds EVERYWHERE
                                 at once. No other code needs to change. Restart Dexter
                                 after flipping.

When EXPLORATION_MODE is True:
  - Trade thresholds: crypto=7, forex=6, stock=6 (vs normal 10/8/8)
  - Max escalation distance from a confluence level: 1.5% (vs normal 0.75%)
  - Weekend crypto threshold bump (+2) is disabled entirely, not just during exploration
    (see CONFLUENCE SCORING SYSTEM section for the reminder-to-restore-on-real-money note)
  - Chev's escalation prompt gets an extra paragraph: this is a paper-trading DATA-
    COLLECTION PHASE, losses are tuition, DEFAULT is POST if there's a nameable entry/
    stop/target, and a SKIP must name ONE concrete broken element (no invalidation level,
    counter-trend without meeting the MTF tax conditions, price nowhere near any zone) —
    never general caution, "not enough confluence", a letter grade alone, or maturity/
    participation figures alone (those are sizing inputs in this phase, not vetoes)
  - Chev's system prompt ("Chev Prompt" file — MUST be manually pasted into Open WebUI
    for this half to actually take effect) carries a matching paragraph right before
    "WHEN DEXTER ESCALATES A SIGNAL": the Operating System's hard skip conditions
    (auction maturity, participation floor, TRANSITIONING state, direction-score
    contradiction) become CONVICTION/SIZING inputs on Dexter escalations ONLY, not vetoes,
    for the duration of the phase. Kev's normal chat requests are completely unaffected.
  - Dexter prints the live regime on startup, right after
    "labeller.start_resolver_daemon(...)" — first screen of the terminal tells you
    whether EXPLORATION MODE is ACTIVE (with its numbers) or OFF (normal thresholds).

DELIBERATELY UNCHANGED BY THE SWITCH (the hygiene stays at full strength — that's the
entire point of this design — none of these read EXPLORATION_MODE at all):
  - STRUCT PRE-GATE and the CONFLICT gate (_directional_split) inside the TRADEABILITY
    PRE-GATE — same sr/vp ≥2 requirement, same bull/bear contradiction check
  - risk_gauntlet.py — ATR SL sanity, R:R ≥ 2.0, position size cap, stale price guard,
    ATR staleness — completely untouched
  - SL-direction hard-rejects
  - _setup_grade() — ALWAYS grades against the FULL normal threshold (10/8/8), regardless
    of exploration mode. This is intentional and is the real risk control of the whole
    phase: a score-7 exploration trade still only grades as a B (or lower), so it gets
    small suggested risk (0.75%) rather than being treated like a full-strength A+ setup.
  - labeller.py, engines.py, honest_sim.py, derivs.py, any jane_* function — all untouched

THE HELPER — _active_thresholds(asset_type) in dexter.py: the single source of truth.
Returns (score_threshold, max_dist_pct) for the given asset type, reading the exploration
constants when EXPLORATION_MODE is True and the normal constants otherwise. Every gate
that checks a score or distance goes through this — never the raw CONFLUENCE_THRESHOLD_*
or ESCALATION_MAX_DIST_PCT constants directly — specifically so the mode can never be
half-applied (e.g. the pre-gate accepting a score-7 setup that the post-Chev hard score
gate then rejects at 10, which is exactly the bug this fixes).

## TRADE FORENSICS (trade_forensics.py, added 2026-07-04)
================================================================================

C:\ChevTools\trade_forensics.py — post-trade path attribution. Pure functions, no
network, no state — same pattern as derivs.py. Self-test: `python -X utf8
trade_forensics.py` → 19/19 passing.

compute_forensics(trade, candles) → dict with EXACTLY these keys: r_dist, mae_r, mfe_r,
bars_held, exit_type_detail, stopped_then_ran, rsi_entry, rsi_exit.
  - r_dist/mae_r/mfe_r require trade["sl_original"] (added at trade creation, see below);
    None for all three if it's missing rather than guessing.
  - exit_type_detail ('wick' vs 'close') and stopped_then_ran (did price reach the
    ORIGINAL tp before re-touching sl_original) are SL-exit-only; stopped_then_ran uses
    the same pessimistic same-candle tie-break as labeller.py (both touched same candle
    → loss wins → False).
  - rsi_entry/rsi_exit are None unless at least 20 candles precede the entry index in the
    supplied candle list (real warm-up, not a garbage Wilder's-RSI value).
  - Internally locates the actual exit candle by walking forward from entry looking for
    the first SL/TP touch — NOT by assuming "last candle in the list", since callers may
    append post-exit candles for the stopped_then_ran check.

Wired into dexter.py:
  1. Trade creation (single call site, right after log_new_trade() in the main escalation
     loop): new_trade["sl_original"]/["tp_original"] stamped alongside the existing
     original_sl/original_tp fields (different field names, deliberately additive —
     original_sl/original_tp already meant something else to honest_sim).
  2. _do_postmortem (fires on every full close): fetches candles via
     _fetch_candles_for_labeller (reused, no new fetch path) at a timeframe keyed by
     trade_type — scalp→5m, day→15m, swing→1h — with a 30-bar buffer before opened_at for
     RSI warm-up. Result stored on the journal entry under the key "forensics". Wrapped so
     any failure logs one line and never blocks the close/journal path.
  3. _run_learning_session: once ≥10 closed entries for an asset type carry a forensics
     dict, a "TRADE PATH FORENSICS (n=X)" block is appended to the learning input (NOT a
     new playbook section) with exactly 5 aggregates: % SL losses classified 'wick', %
     SL losses where stopped_then_ran, median MFE of losers, median MAE of winners,
     average bars held wins vs losses. Below 10 samples: nothing appended.

## DEXTER SCAN STRINGS — EXACT FORMATS (important for REASON_MAP)
================================================================================

These are the verbatim strings dexter's scan_pair_tf() puts in result["reasons"].
The Examiner's normalize_reasons() maps these to canonical tokens. Knowing the
exact format prevents substring pollution (e.g. "EMA55 support" → ema55, not sr).

  SR:        "Resistance(3x,3pt)"  "Support(2x,2pt)"   format: {Label}({N}x,{pts}pt)
             Resistance = sr_multi if N>=3, sr_single if N<=2. Same for Support.
  Fib:       "Fib 61.8% (golden pocket) (2pt)"  "Fib 50% (2pt)"  "Fib 38.2%"
  GP:        "★ GOLDEN POCKET"
             "★★ GP×SR DEADLY COMBO (×1.15 quality bonus applied)"  → gp_sr_combo
  GP watch:  "[WATCH — approaching GP] GP zone [...]..."  → gp_approach (not scored)
  RSI div:   "Regular Bearish Divergence (str=1.5pt)"  "Hidden Bullish (str=1.0pt)"
             "Hidden Bearish (str=1.0pt)"  "Regular Bullish Divergence (str=1.0pt)"
  Forming:   "[WATCH — not yet confirmed] FORMING bullish divergence (RSI gap Xpt, Y bars, str=Zpt)"
             → rsi_div_forming (NEVER maps to rsi_div — separate stat bucket)
  EMA:       "EMA55 support (2.0pt)"  "EMA21 resistance (1.0pt)"  "EMA13 support (0.5pt)"
             format: "{ema_label} {side} ({weight}pt)"
  EMA cross: "EMA crossover bullish_cross 2c ago"  → ema_cross
  BB:        "BB upper BURST (%B=1.12, 12.0% of band outside, 2pt)"  → bb_burst
             "BB lower BURST (%B=-0.05, ...)"                         → bb_burst
             "BB near upper (%B=0.88, top 15% of band, 0.5pt)"       → bb_near
             "BB near lower (%B=0.12, ...)"                           → bb_near
             "BB mid support (%B=0.50, 1pt)"                          → bb_mid
             "BB mid resistance (%B=0.50, 1pt)"                       → bb_mid
             "BB squeeze (width 1.2%, context, 0pt)"                  → bb_squeeze (0pt since 2026-07-04 — see CONFLUENCE SCORING SYSTEM)
  VP:        "VP POC 4h (0.15% away ≤0.40% prox, 3pt)"               → vp_poc
             "VP VAH 1h (...)"  → vp_vah     "VP VAL 4h (...)"  → vp_val
             Unconfirmed-anchor variant (half points): "VP POC 4h (0.12% away ≤0.40%

## RISK GAUNTLET (risk_gauntlet.py)
================================================================================

C:\ChevTools\risk_gauntlet.py — Pre-trade risk validation, 13/13 tests passing.

Called by Dexter immediately after Chev posts a TRADE: line. If any gate fails,
Dexter rejects the trade with a logged reason (no Google Sheets entry).

Gates (in order):
  1. ATR SL sanity: SL must be between 0.5× and 4× ATR from entry.
       < 0.5× ATR → RED zone (too tight, will be stopped by noise)
       > 4× ATR   → RED zone (position size will be negligibly small)
       0.5–1×     → YELLOW zone (tight — Chev warned but not blocked on this alone)
  2. R:R minimum: TP must be ≥ 2.0× the SL distance from entry. Flat 2.0 for all grades.
  3. Position size cap: position_size_usd ≤ account_balance (no over-leveraging).
       Only fires when balance > 0 (guards against Google Sheets read failure).
  4. Stale price guard: entry price must be within 1.5% of live price (prevents
       fills on completely stale Chev decisions made while price moved away).
  5. ATR staleness: ATR computed from candles ≤ 6 hours old (scalp/day) or 24h (swing).

Cost constants (used by labeller for cost_R calculation — must stay in sync):
  Fee (per side):      crypto=0.05%  forex=0.015%  stock=0.02%
  Slippage (per side): crypto=0.02%  forex=0.01%   stock=0.02%
  Overnight funding (swing only): 0.06% per side

R multipliers (must match labeller.py R_MULT):
  scalp=1.0×ATR  day=1.2×ATR  swing=1.5×ATR

## THE EXAMINER (labeller.py)
================================================================================

C:\ChevTools\labeller.py — Triple-barrier labelling + meta-labelling. 26/26 tests passing.

Purpose: Build a shadow trading record for EVERY setup Dexter encounters (POST, SKIP,
NOT_ESCALATED, REJECT), label each outcome objectively, and measure whether Chev's
judgment adds alpha above the raw Dexter score.

--- SHADOW RECORD LIFECYCLE ---

Stage 1: PENDING — created when Dexter encounters a setup.
  - entry_ref = result["price"] (the confluence zone), NOT current_price.
    Using current_price would collapse the two-stage design (immediate touch).
  - expiry_ts = now + EXPIRY_HOURS[trade_type] from setup time.
  - Waits for price to touch the zone (touch_band = max(0.1%, 0.25×ATR)).
  - If price never touches by expiry → label = NO_FILL (closed as untraded).

Stage 2: ACTIVE — armed when zone is touched.
  - active_expiry_ts = touched_ts + EXPIRY_HOURS[trade_type] from touch time.
  - session_at_touch and regime_at_touch stamped at transition.
  - Barriers armed: upper (profit), lower (loss), upper_2r (+2R target).
  - Long:  upper = entry+1R  lower = entry-1R  upper_2r = entry+2R
  - Short: upper = entry-1R  lower = entry+1R  upper_2r = entry-2R

Labels:
  +1     — profit barrier hit before loss barrier
  -1     — loss barrier hit (or same-candle pessimistic rule fires)
   0     — active_expiry reached without barrier; realized_R = drift (close−entry)/r_dist
  NO_FILL — PENDING expired without touch (never activated)
  VOID    — ACTIVE expiry passed 7+ days ago with no candle data covering the window.
             Honest: we refuse to invent a label when data is unavailable.

Pessimistic same-candle rule: if BOTH barriers hit in the same candle → label=-1.
Rationale: worst-case assumption; eliminates fill-order ambiguity.

hit_2r flag (post-resolution trailing walk):
  Definition: "did price touch +2R before returning to the loss barrier AFTER profit exit?"
  After label=+1 resolves, walk subsequent candles (strictly after profit exit candle):
    - +2R reached before loss barrier → hit_2r=True
    - Loss barrier reached before +2R → hit_2r=False
    - Same candle: both 2R and loss → loss wins, hit_2r=False (pessimism)
    - active_expiry or no subsequent candles → hit_2r=False
  bars_held is frozen at the profit exit candle (trailing walk is post-resolution only).

EXPIRY_HOURS (wall-clock, not bars):
  scalp=2h  day=6h  swing=48h

Deduplication:
  record_setup() skips if an unresolved PENDING record for the same symbol+tf already
  has entry_ref within 0.5×ATR. Prevents 2-minute rescan loop from flooding the shadow
  book with near-identical records for the same persistent zone.

--- DIRECTION SOURCE ---

direction_src = "chev"  — direction came from Chev's TRADE: line (POST decision).
direction_src = "proxy" — direction inferred from zone position:
    result["price"] (zone) < result["current_price"] → zone below price → support → long
    result["price"] > result["current_price"]        → zone above price → resistance → short
    At-level (within 0.05%): regime tiebreak (TRENDING_UP → long, TRENDING_DOWN → short).

CRITICAL: chev and proxy stats are NEVER mixed in any report. Separate buckets always.
This enforces statistical integrity: chev direction reflects Chev's active judgment;
proxy direction reflects a structural baseline. Mixing the two poisons both.

--- EFFECTIVE N (overlap weighting) ---

When multiple records share asset_type + direction with overlapping ACTIVE windows,
they are statistically non-independent (same market move affects all of them).
Effective N = sum(1/k) where k = count of overlapping co-directional records.
All report stats use eff_N, not raw_N. This prevents spurious high confidence from
correlated trades made during the same trending move.

--- REPORT FUNCTIONS ---

chev_scoreboard()   — POST vs SKIP expectancy stratified by Dexter score band.
                      Bands: [10-12], [12-14], [14+]. Wilson 95% CI on all win rates.
                      Net expectancy = realized_R - cost_R (round-trip trading cost).

tag_lift_report()   — Per-feature-token lift stats.
                      lift = P(win|token) / P(win overall, base rate).
                      Temporal discovery split: discovery_ts = first epoch where token
                      SIMULTANEOUSLY meets eff_N ≥ 30 AND lift > 1.0 (using running
                      base rate at that point — no lookahead). Persisted in
                      labels_discovery.json once set, never revised.
                      Post-discovery actionable = eff_N ≥ 30 AND CI_lo > post-discovery base rate.

weight_suggestions()  — For tokens with eff_N ≥ 50: INCREASE / DECREASE / KEEP.
                        INCREASE: lift > 1.5 AND CI_lo > base_rate.
                        DECREASE: lift < 0.8.

skip_reason_audit()   — Groups closed records by chev_decision:chev_reason.
                        Shows win rate + Wilson CI + avg_R per gate cause.
                        If NOT_ESCALATED cooldown setups outperform POST → gate is hurting.

fill_rate_report()    — fill_rate = filled / (filled + NO_FILL). VOID excluded from both.

full_report()         — Writes all the above to label_report.txt.

## THE HONEST SIMULATOR (honest_sim.py) — Steps 1-2 of 4 done
================================================================================

C:\ChevTools\honest_sim.py — candle-true exit simulation, real trading costs, and a
daily circuit breaker. Self-test: 25/25 passing (`python -X utf8 honest_sim.py`).

PURPOSE: today's paper-trade exits are checked against ~2-minute spot polls, which
(a) misses wicks that pierce SL and recover between polls, (b) checks TP before SL when
both were touched (optimistic ambiguity), (c) fills exactly at TP/SL even through gaps,
and (d) — before this build — left one close path (_startup_sl_audit) charging zero fees.
This makes the shadow book (labeller.py, which already grades pessimistically) a fairer
referee than the real trade log, corrupting POST-vs-SKIP comparisons.

CORRECTION TO PRIOR HANDOFF: this build's research found the "MILESTONE PARTIAL TP +
CHEV JUDGMENT" feature (previously listed above as "designed but not built") was ALREADY
LIVE in dexter.py — _ask_chev_partial_tp() and _execute_partial_close() implement exactly
that HOLD/TAKE25/TAKE50/TRAIL_SL flow at trade-type thresholds (scalp=0.7R, day=1.0R,
swing=1.5R). Also already live before this build: cost deduction on most close paths via
_sim_trading_cost(), and r_multiple already computed/journaled in _do_postmortem. The
honest_sim project's real net-new work is narrower than first scoped: candle-true wick/gap
detection for forex+stock (crypto already had a cruder version via _detect_sl_tp_hit),
candle-true PENDING fills for ALL asset types (previously spot-only everywhere), and the
circuit breaker (fully new).

--- STEP 1: honest_sim.py module (done) ---

Public functions: get_exit_candles, check_exit, check_pending_fill, apply_costs,
record_close_R, breaker_status. Config mirrors risk_gauntlet.FEE_SIDE/SLIPPAGE_SIDE/
FUNDING_EST_SWING (imported directly, not duplicated).

THE RULEBOOK (check_exit, applied per-candle in order — this is law, not a heuristic):
  R1 gap-through-SL: candle open already at/beyond SL → exit at OPEN (worse than SL).
  R2 SL touch: checked BEFORE any profit logic — ambiguous candle (both SL+TP touched)
      always resolves SL_HIT (house pessimism).
  R3 partial at 1R: fires once, only if not already partial_done/sip_active and
      orig_tp_rr > 1.05. Same-candle ordering: partial first, then TP if also touched.
  R4 gap-through-TP: open already at/beyond TP → exit at TP exactly (never better).
  R5 TP touch.
  R6 nothing touched → next candle.

check_exit has a THREE-STATE return (all callers must branch on
`result is not None and result["exit_price"] is not None`, never bare dict truthiness,
since a partial-only result is a truthy dict with exit_price/close_type = None):
  None                                    → nothing happened
  {exit_price: None, close_type: None, partial: {...}}  → 1R partial fired, still open
  {exit_price: float, close_type: str, partial: dict|None}  → trade exited

check_pending_fill: STOP-style orders (breakout, chasing the trigger direction) fill at
the candle OPEN if it already gapped past entry (worse than quoted); LIMIT-style orders
(pullback, against the trigger direction) always fill at exactly entry, never better,
even on a favorable gap. The remainder of that same fill candle is then run through
check_exit using its REAL open (not the fill price) — a fill candle can gap you straight
through your stop in the same bar, and that must surface as an exit at the real gap price.

get_exit_candles: returns None on genuine fetch failure (caller falls back to spot logic)
vs [] on a successful fetch with nothing new yet (normal, most cycles — NOT a fallback
trigger). Crypto uses a small dynamic candle count (1m, no scanner equivalent to share a
cache with). Forex/stock request a FIXED 500 candles matching the main scanner's own 15m
fetch depth (tf_fetch_limits in dexter.py) specifically so whichever of the two callers
runs first this cycle populates fetch_candles()'s 60s cache for the other — this only
works because the request SIZE matches; a dynamically-sized request (what was tried first)
almost never matches the scanner's size and would just add a second network call.

--- STEP 2: dexter.py wiring (done) ---

Anchor A: `import honest_sim` added.
New helper _honest_sim_last_check_ts(trade): reads trade["last_exit_check_ts"], else
parses trade["opened_at"] (the reliable UTC string field — see KNOWN BUGS re: open_ts
vs opened_at), else falls back to now-300s. Legacy trades from before this deploy get
bootstrapped automatically, no backfill needed.

Anchor C (PENDING, check_and_update_open_trades): candle fetch failure → byte-identical
spot fallback. Otherwise honest_sim.check_pending_fill decides the fill. Fill price is
now written back into trade["entry"] AND sheet column C (can legitimately differ from the
quoted entry on a stop-style gap fill). If the fill candle also produced an immediate
SL/TP hit, the trade opens and closes in the same cycle.

Anchor B (OPEN, same function): fetch failure → the ENTIRE original spot block (TP/SL
check + legacy 1R auto-partial) runs completely unchanged, still calling
_sim_trading_cost — that swap is Step 3's job, not Step 2's. Otherwise honest_sim.check_exit
decides; a partial executes at the exact 1R price via apply_costs(fraction=0.5) and writes
the same-shaped PARTIAL_1R journal entry as before; a real exit still computes cost via
_sim_trading_cost for now (same reason). The legacy partial block gained one guard clause
(`exit_candles is None and ...`) so it only fires in the fallback branch. Milestones, SIP
transition/ratchet, swing trailing, BOS alerts: untouched, still spot-price-driven, still
unconditional.

Jane (jane_*, check_and_update_jane_trades): zero edits, confirmed untouched — deliberate
control arm on the old rules.

## KNOWN BUGS / GOTCHAS
================================================================================

- client.session.timeout in gspread: NEVER add this. That version of gspread has no
  .session attribute on Client. Adding it causes AttributeError.

- Expo Node version: Node 24 breaks Expo config plugins. Always `nvm use 20` first.

- Dexter's /api/pending: always guard with `if worksheet is None: return jsonify([])`
  before any worksheet.get_all_values() call. Worksheet can be None when Sheets fails.

- Firebase SSE race: RSI div lines (_rsi_div flag) must NEVER be saved to Firebase/
  localStorage. _clearRsiDivOverlay() and _drawRsiDivVisual() must never call
  saveDrawings(). SSE listener must preserve live _rsi_div drawings on update.

- labeller.py: direction_src "chev" and "proxy" must NEVER be mixed in statistics.
  A setup where Chev's direction contradicts the zone proxy is NOT an error — it's
  Chev exercising judgment. Record both; compare separately.

- labeller.py entry_ref for non-POST: always result["price"] (the confluence zone).
  Never result["current_price"] — that would make every record immediately touch-activate.

- Two different "when opened" fields exist on a trade dict: opened_at (a plain UTC
  string, "%Y-%m-%d %H:%M:%S", reliably stamped at creation for every real trade) and
  open_ts (an ISO-format field _sim_trading_cost() reads for crypto-swing funding math,
  but which is basically never populated for real trades — only ever set when loading
  old Jane rows from the sheet). Net effect: crypto swing funding cost has likely been
  silently computing as $0 this whole time. honest_sim keys off opened_at, not open_ts.
  Not fixed as part of the honest_sim build (out of scope) — flagging for whenever it is.

- "Chev Prompt" is the ACTUAL filename — no underscore, no file extension. Task briefs
  and prior conversation sometimes refer to it as "Chev_Prompt" or "chev_prompt.txt";
  those don't exist. Editing the real file changes nothing in Chev's behavior until the
  content is manually pasted into Open WebUI's model system prompt editor.

- _setup_grade()'s min_score must NEVER be routed through _active_thresholds() — it is
  the one deliberate exception to "every gate reads thresholds through the helper".
  Grading always uses the full normal per-asset threshold regardless of EXPLORATION_MODE,
  because the grade (and its suggested risk %) is the actual risk control during the
  exploration phase. If a future edit "fixes" this to be consistent with the other
  threshold call sites, it defeats the entire safety design of exploration mode.

- chev_playbook.txt (no asset suffix, legacy) is frozen but NOT dead: _run_learning_
  session is never called with asset_type=None anymore, so nothing rewrites this file —
  but the Firebase monitor snapshot push and _ask_chev_about_jane_trade() both still read
  it live. Don't assume "no writer" means "safe to delete."
